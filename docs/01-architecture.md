# Pactura System Architecture v1

## Overview

Pactura is designed as a multi-tenant, cloud-native system deployed on Google Cloud Platform.  
The architecture prioritizes strict tenant isolation, role-based access control, and governed document handling.

The system follows a layered model:

1. Presentation Layer  
2. API Enforcement Layer  
3. Data and Storage Layer  
4. Automation and Governance Layer  


---

## Component Responsibilities

### 1. Presentation Layer – Next.js Web App

Responsibilities:
- User interface and routing
- Firebase Authentication integration
- Organization selection
- Document upload interface
- Chat interface with citation rendering

The frontend never directly accesses databases or raw storage.  
All operations flow through the API layer.

---

### 2. API Enforcement Layer – Cloud Run

This is the architectural control plane.

Responsibilities:
- Verify Firebase ID tokens
- Resolve organization membership
- Enforce role-based access control (RBAC)
- Enforce tenant isolation via org_id scoping
- Handle document ingestion and chunking
- Generate embeddings
- Execute vector retrieval queries
- Orchestrate LLM calls
- Write audit events
- Update usage counters
- Generate signed URLs for secure downloads

All data-layer access is routed through this service.

---

### 3. Data and Storage Layer

#### Firestore

Purpose:
- Organization metadata
- Membership and roles
- Invites
- Audit logs
- Usage counters
- Chat session metadata

Firestore is not trusted for authorization alone.  
RBAC enforcement occurs in the API layer.

---

#### Cloud SQL (PostgreSQL + pgvector)

Purpose:
- Store document metadata
- Store chunked document text
- Store vector embeddings
- Execute tenant-scoped similarity search

Every query includes an org_id filter to prevent cross-tenant leakage.

---

#### Cloud Storage

Purpose:
- Store raw uploaded documents
- Optionally store derived artifacts

The bucket is private.  
Access is granted only through short-lived signed URLs generated by the API.

---

#### Secret Manager

Purpose:
- Store database credentials
- Store LLM provider API keys
- Store email provider keys

No secrets are committed to the repository.

---

### 4. Automation and Governance Layer

#### Cloud Scheduler

Purpose:
- Trigger daily retention sweep
- Enforce document expiration policies
- Initiate hard-delete workflows

Retention logic is executed via a protected API route.

---

## Multi-Tenancy Model (Summary)

Pactura uses logical multi-tenancy.

Isolation is enforced at:
- API request validation
- Firestore document paths
- SQL queries via org_id scoping
- Vector similarity search filters

No tenant data is accessible without explicit org context validation.

---

## Design Priorities

- Single enforcement boundary
- Explicit tenant scoping
- Secure default configuration
- Cost-controlled infrastructure
- Modular evolution path
